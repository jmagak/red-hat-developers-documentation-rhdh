= Trace attributes reference for filtering and querying workflows

[role="_abstract"]
Automatically generated span attributes and lifecycle events including process IDs, workflow states, and execution milestones. Reference this data dictionary to build Jaeger queries that filter by workflow name, find failed executions, or track requests across service boundaries.

This data allows you to track a workflow from start to finish, analyze external function calls, and correlate logs across asynchronous boundaries.

== Filter by span attributes

To locate specific workflow executions or trace a process through various states, use the automatic span attributes generated by SonataFlow. Each span includes the following specific attributes:

* `sonataflow.process.id`: The definition ID of the workflow.
* `sonataflow.process.instance.id`: The unique ID for the specific execution instance.
* `sonataflow.process.version`: The workflow definition version.
* `sonataflow.workflow.state`: The name of the current workflow state, for example, `StartEvent`.
* `sonataflow.process.instance.state`: The current state of the process instance, such as `ACTIVE`, `COMPLETED`, `ERROR`, or `SUSPENDED`.
* `sonataflow.transaction.id`: The ID used to correlate multiple workflows in one business transaction.
* `sonataflow.tracker.*`: Custom attributes converted from `X-TRACKER-*` headers.
* `service.name` and `service.version`: The service identification details from your configuration.

== Track the timeline using process lifecycle events

To understand the exact chronological order of a workflow's execution, look for OpenTelemetry events that are automatically generated at key lifecycle points. SonataFlow attaches the following events and their specific attributes to spans to mark execution milestones:

* `process.instance.start`: Marks the beginning of the execution. It includes the `process.instance.id`, what `trigger` started it, and the `reference.id`.
* `process.instance.complete`: Marks the end of the workflow. It includes the `process.instance.id`, the final `outcome`, and the total `duration.ms`.
* `process.instance.error`: Added when a workflow fails. It includes the `process.instance.id`, the `error.message`, and the `error.type`.
* `state.started` and `state.completed`: Generated for each individual workflow state. Both include an `event.description` detailing the state execution.
* `log.message`: Includes the actual application log content within the trace span. It provides the `level`, `logger`, `message`, `thread.name`, and `thread.id`.

== Debug external integrations with function call attributes

When a workflow invokes an external function, for example, a REST service, the function call spans are enriched with additional HTTP attributes:

* `sonataflow.function.name`: The name of the function being called.
* `sonataflow.function.type`: The type of function, such as `rest` or `expression`.
* `http.method`: The HTTP method used for external REST calls.
* `http.url`: The full target URL for the external call.
* `http.status_code`: The resulting HTTP response code from the service.

== Trace context propagation across services

SonataFlow extracts and propagates the following headers to maintain observability across service boundaries:

* `X-TRANSACTION-ID`: Correlates multiple workflow executions that belong to the same broader business transaction.
* `X-TRACKER-*`: Any custom tracking context provided in these headers is sanitized and automatically converted into span attributes, for example, `sonataflow.tracker.*`, for easier querying.
