= Platform compatibility and supported configurations

[role="_abstract"]
To ensure compatibility and avoid collection failures, you can verify platform support and determine the correct collection method by reviewing requirements, deployment coverage, and version compatibility for {ocp-short}, {aks-short}, {eks-short}, and {gke-short}.

For best results, exclude collectors not relevant to your deployment method.
For example, Helm-based deployments should exclude the Operator collector, and Operator-based deployments should exclude the Helm collector.

== {product} version compatibility

The {product-very-short} must-gather tool version should match your {product-very-short} deployment version for compatibility with deployment methods, resource schemas, and platform features.

.{product} version alignment
[cols="1,1,2", options="header"]
|===
|{product} Version
|must-gather Image Tag
|Notes

|1.10.0
|`1.10.0`
|First release with must-gather as {technology-preview}. Supports Helm and Operator collectors.

|1.9.x and earlier
|Not available
|Must-gather not available. Use manual collection with `oc logs`, `oc get`, and `helm get` commands.

|Latest
|`latest`
|Tracks the most recent {product-very-short} release. Use for latest features and fixes.
|===

Backward compatibility considerations:

* Must-gather images can collect data from older {product-very-short} deployments, but may not recognize newer resource types or features.
* For best results, use a must-gather image version that matches or is slightly newer than your {product-very-short} deployment.
* If your deployment spans multiple {product-very-short} versions (for example, multiple namespaces with different versions), use the must-gather image version that matches the highest {product-very-short} version.

== Disconnected and air-gapped environment support

The {product-very-short} must-gather tool supports disconnected environments with image mirroring.

Disconnected environment requirements:
* Mirror the must-gather image to your internal container registry
* Configure must-gather to use the mirrored image location
* Ensure network connectivity from the cluster to the internal registry

.Mirroring the must-gather image
[source,terminal]
----
$ podman pull quay.io/rhdh-community/rhdh-must-gather:1.10.0
$ podman tag quay.io/rhdh-community/rhdh-must-gather:1.10.0 internal-registry.example.com/rhdh/must-gather:1.10.0
$ podman push internal-registry.example.com/rhdh/must-gather:1.10.0
----

.Using the mirrored image on OpenShift
[source,terminal]
----
$ oc adm must-gather --image=internal-registry.example.com/rhdh/must-gather:1.10.0
----

.Using the mirrored image on Kubernetes (Helm values)
[source,yaml]
----
image:
  repository: internal-registry.example.com/rhdh/must-gather
  tag: "1.10.0"
----

Data transfer considerations for disconnected environments:

* Collected diagnostic data must be transferred from the disconnected environment to a system with internet access for support case upload
* Output directory size varies based on collection scope (see xref:diagnostic-data-output-structure-and-organization_{context}[])
* Heap dumps significantly increase output size (500 MB - 2 GB total)
* Test collection scope in a non-production environment to estimate output size before production use

== Required RBAC permissions

The {product-very-short} must-gather tool requires elevated permissions to read cluster resources, pod logs, and namespace information.

Minimum required permissions:
* Read access to pods, deployments, services, configmaps in target namespaces
* Read access to pod logs
* Read access to custom resources ({product-custom-resource-type} CRs for Operator deployments, Helm releases for Helm deployments)
* Read access to routes ({ocp-short}) or ingress resources (Kubernetes)
* Read access to secrets (only when using `--with-secrets`)
* Ability to execute commands in pods (only when using `--with-heap-dumps` to send signals)

On {ocp-short}, the `oc adm must-gather` command typically runs with cluster-admin privileges, which include all necessary permissions.

On Kubernetes platforms, the must-gather Helm chart creates a service account with appropriate role bindings.
If your cluster has restrictive security policies, you may need to create additional role bindings or use a service account with elevated permissions.

== Known limitations and unsupported scenarios

The following scenarios are not currently supported or have known limitations:

Unsupported platforms::
Kubernetes distributions other than {aks-short}, {eks-short}, {gke-short}, and {ocp-short} are not tested or officially supported.
The tool may work on other Kubernetes distributions, but functionality is not guaranteed.

Multi-cluster deployments::
The must-gather tool collects data from a single cluster.
For {product-very-short} deployments spanning multiple clusters, run must-gather separately on each cluster.

Dynamic plugin diagnostics::
While the tool collects pod logs and configurations that include dynamic plugin state, it does not have specialized collectors for individual dynamic plugins.
Plugin-specific diagnostics rely on application logs and `ConfigMap` data.

Ephemeral environments::
The must-gather tool assumes resources remain stable during collection (2-15 minutes depending on scope).
In highly dynamic environments with frequent pod restarts, some data may be incomplete.

== Platform-specific implementation notes

{ocp-short}::
The must-gather tool integrates with the OpenShift must-gather framework.
Collection runs in a dedicated pod created by the `oc adm must-gather` command.
The pod is automatically cleaned up after collection completes.
Output is saved to the local filesystem in the `must-gather.local.<timestamp>` directory.

{aks-short}::
The must-gather tool deploys as a Kubernetes Job via Helm chart.
Collection runs to completion, and the output must be extracted from the pod using `kubectl cp`.
The job and pod resources are retained after completion to allow data extraction, then manually cleaned up using `helm uninstall`.

{eks-short}::
Implementation is identical to {aks-short}.
The Helm chart creates a Job that runs must-gather collection.
Authentication to the {eks-short} cluster uses `kubectl` with {aws-short} credentials.

{gke-short}::
Implementation is identical to {aks-short}.
The Helm chart creates a Job that runs must-gather collection.
Authentication to the {gke-short} cluster uses `kubectl` with GCP credentials.
