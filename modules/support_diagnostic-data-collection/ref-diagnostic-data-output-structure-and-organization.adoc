= Diagnostic data output structure and organization

[role="_abstract"]
To quickly locate diagnostic information and reduce analysis time, you can navigate the output directory structure organized by collector type, namespace, and resource type for logs, configurations, and resource definitions.

== Top-level directory structure

The must-gather output directory contains subdirectories organized by collection method and data type.

.Top-level output directory organization
[cols="1,3", options="header"]
|===
|Directory
|Contents

|`helm/`
|Helm release information including release metadata, deployed values, and generated manifests. Present only if the Helm collector ran.

|`operator/`
|Operator-specific data including operator logs, `CRDs`, and {product-custom-resource-type} custom resources. Present only if the Operator collector ran.

|`orchestrator/`
|Workflow execution data, {rhoserverless-brand-name} resources, and Knative state. Present only if the Orchestrator collector ran.

|`namespaces/`
|Per-namespace diagnostic data organized by namespace name. Contains subdirectories for each namespace where resources were collected.

|`cluster-info/`
|Cluster-wide state information if `--cluster-info` was specified. Contains cluster-level resources and configurations.

|`platform/`
|Platform detection metadata, cluster version, node information, and storage classes.

|`routes/`
|OpenShift route definitions. Present only on {ocp-short} clusters if the route collector ran.

|`ingress/`
|Kubernetes ingress configurations. Present on Kubernetes platforms if the ingress collector ran.

|`sanitization-report.txt`
|Report of sanitized secret fields if `--with-secrets` was specified. Lists secrets that were processed and fields that were redacted.

|`metadata.json`
|Collection metadata including timestamps, must-gather version, collectors that ran, and collection parameters.
|===

== Namespace directory structure

Within the `namespaces/` directory, each namespace has a subdirectory containing resource definitions and logs organized by Kubernetes resource type.

.Per-namespace directory organization
[cols="1,3", options="header"]
|===
|Subdirectory
|Contents

|`namespaces/<namespace-name>/pods/`
|Pod definitions and logs. Each pod has a subdirectory containing its YAML definition and log files for each container.

|`namespaces/<namespace-name>/deployments/`
|Deployment resource definitions in YAML format.

|`namespaces/<namespace-name>/services/`
|Service resource definitions.

|`namespaces/<namespace-name>/configmaps/`
|`ConfigMap` resources containing application configuration.

|`namespaces/<namespace-name>/secrets/`
|Kubernetes secrets with sanitized values (present only if `--with-secrets` was specified).

|`namespaces/<namespace-name>/persistentvolumeclaims/`
|Persistent volume claim definitions.

|`namespaces/<namespace-name>/routes/`
|Namespace-scoped route resources ({ocp-short} only).

|`namespaces/<namespace-name>/ingress/`
|Namespace-scoped ingress resources (Kubernetes platforms).

|`namespaces/<namespace-name>/<custom-resources>/`
|Custom resource definitions specific to {product-very-short} and installed plugins.
|===

== Log file naming conventions

Pod logs follow a consistent naming pattern to identify the pod and container that generated each log file.

.Log file naming pattern
----
namespaces/<namespace-name>/pods/<pod-name>/logs/<container-name>.log
----

For example:
----
namespaces/rhdh-prod/pods/backstage-developer-hub-7d8f9c5b-xk2m4/logs/backstage-backend.log
----

If a pod has multiple containers, each container has a separate log file:
----
namespaces/rhdh-prod/pods/backstage-developer-hub-7d8f9c5b-xk2m4/logs/backstage-backend.log
namespaces/rhdh-prod/pods/backstage-developer-hub-7d8f9c5b-xk2m4/logs/oauth2-proxy.log
----

== Helm collector output

When the Helm collector runs, it creates a directory structure containing release information.

.Helm collector output structure
[cols="1,3", options="header"]
|===
|Path
|Contents

|`helm/releases.txt`
|List of all Helm releases in the cluster with release names, namespaces, chart names, and statuses.

|`helm/<release-name>/values.yaml`
|Deployed values for the Helm release.

|`helm/<release-name>/manifest.yaml`
|Full rendered manifest showing all resources generated by the Helm chart.

|`helm/<release-name>/metadata.json`
|Helm release metadata including chart version, app version, and release timestamps.
|===

== Operator collector output

When the Operator collector runs, it gathers operator-specific resources.

.Operator collector output structure
[cols="1,3", options="header"]
|===
|Path
|Contents

|`operator/operator-logs/`
|Logs from the {product-very-short} Operator pod.

|`operator/crds/`
|Custom resource definitions installed by the Operator.

|`operator/backstage-crs/`
|{product-custom-resource-type} custom resource instances showing operator-managed {product-very-short} deployments.

|`operator/operator-config/`
|Operator configuration including `RBAC` resources and service accounts.
|===

== Heap dump file locations

When heap dumps are collected, they appear within pod directories in the namespace structure.

.Heap dump file naming pattern
----
namespaces/<namespace-name>/pods/<pod-name>/heapdumps/<timestamp>.heapsnapshot
----

For example:
----
namespaces/rhdh-prod/pods/backstage-developer-hub-7d8f9c5b-xk2m4/heapdumps/20260430-143022-1.heapsnapshot
----

Heap dump files are binary files that can be analyzed using Chrome DevTools, Node.js heap analysis tools, or specialized memory profilers.

== Output directory size estimates

The size of the must-gather output depends on collection scope and optional data inclusion.
These are approximate estimates for typical deployments.
Actual sizes vary based on log volume, number of pods, and retention period.
Test in a non-production environment to estimate size for your deployment.

.Typical output sizes by collection scope
[cols="1,1,2", options="header"]
|===
|Collection Scope
|Typical Size
|Notes

|Basic collection (default)
|10-50 MB
|Platform info, deployments, configs, pod logs for small to medium deployments

|With cluster-info
|50-100 MB
|Adds cluster-wide resource definitions

|With secrets (sanitized)
|15-60 MB
|Adds secret YAML files, minimal size increase

|With heap dumps
|500 MB - 2 GB
|Each heap dump is 50-500 MB depending on backend memory usage

|Multi-namespace large deployment
|100-200 MB
|Without heap dumps, 10+ namespaces with multiple {product-very-short} instances
|===

== Using OpenShift Must-Gather Client for analysis

The OpenShift Must-Gather Client (OMC) is a command-line tool that simplifies analysis of must-gather output.
OMC provides query capabilities and structured views of collected data.

.Installing OMC
[source,terminal]
----
$ python3 -m pip install --user omc
----

.Loading must-gather data into OMC
[source,terminal]
----
$ omc use must-gather.local.<timestamp>
----

Common OMC queries for {product-very-short} diagnostics:

List all pods in the must-gather output:
[source,terminal]
----
$ omc get pods
----

Show pod logs for a specific pod:
[source,terminal]
----
$ omc logs <pod-name>
----

List all `ConfigMaps`:
[source,terminal]
----
$ omc get configmaps
----

Filter resources by namespace:
[source,terminal]
----
$ omc get pods -n rhdh-prod
----

Search logs for specific errors:
[source,terminal]
----
$ omc logs -A | grep -i "error"
----

OMC provides a more efficient interface for navigating large must-gather outputs compared to manual file system navigation.

== Locating common diagnostic data

The following table shows where to find frequently needed diagnostic information in the must-gather output.

.Common diagnostic data locations
[cols="1,2", options="header"]
|===
|Diagnostic Data
|Location

|Backend pod logs
|`namespaces/<namespace>/pods/<backstage-pod>/logs/backstage-backend.log`

|PostgreSQL connection configuration
|`namespaces/<namespace>/secrets/postgres-credentials.yaml` (if secrets collected) or `namespaces/<namespace>/configmaps/<app-config-configmap>.yaml`

|Operator logs
|`operator/operator-logs/<operator-pod>/logs/manager.log`

|{product-custom-resource-type} custom resource definition
|`operator/backstage-crs/<backstage-cr-name>.yaml`

|Route/Ingress definitions
|`routes/<route-name>.yaml` ({ocp-short}) or `ingress/<ingress-name>.yaml` (Kubernetes)

|Helm release values
|`helm/<release-name>/values.yaml`

|Deployment configurations
|`namespaces/<namespace>/deployments/<deployment-name>.yaml`

|Service definitions
|`namespaces/<namespace>/services/<service-name>.yaml`

|Platform and cluster version
|`platform/cluster-version.yaml` or `platform/nodes.yaml`

|Heap dumps for memory analysis
|`namespaces/<namespace>/pods/<pod-name>/heapdumps/*.heapsnapshot`
|===
