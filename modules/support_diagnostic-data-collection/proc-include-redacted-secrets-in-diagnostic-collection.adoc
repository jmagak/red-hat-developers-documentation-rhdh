= Include redacted secrets in diagnostic collection

[role="_abstract"]
When troubleshooting configuration issues, you can include Kubernetes secrets in diagnostic data collection with automatic redaction to minimize security risk while preserving structure and keys needed for analysis.

.Prerequisites

* Understanding of basic `must-gather` collection on your platform.
* Knowledge of which configuration issues require seeing secret structure.
* Cluster administrator access (secrets collection requires elevated permissions).
* Approval from your security team if required by your organization's policies.

.Procedure

. Assess whether secrets collection is necessary for your troubleshooting scenario.
Secrets collection is useful when:
+
. Debugging configuration issues related to external integrations (database connections, authentication providers, API credentials)
. Troubleshooting permission errors that mightbe caused by incorrect secret references or malformed secret data
. Verifying that secret keys match the expected configuration schema
+
Do not include secrets if you are troubleshooting issues unrelated to configuration, such as performance problems or resource constraints.

. Understand the security considerations:
+
. The tool redacts sensitive values (passwords, tokens, API keys, certificates) using pattern matching on field names (for example, `password`, `token`, `apiKey`) and value formats (for example, API key patterns)
. The tool preserves secret keys and structure to enable configuration troubleshooting
. The tool decodes Base64-encoded values in secret `data` fields before pattern matching to detect encoded sensitive data
. A redaction report documents which fields the tool redacted
. Manually review the output before sharing it with support to verify no sensitive data remains
+
[IMPORTANT]
====
The tool might not detect Base64-encoded values in fields without sensitive names.
Fields named generically (for example, `value`, `content`) that contain encoded secrets require manual verification.
====

. Enable secrets collection:
+
--
On {ocp-short}:
[source,terminal]
----
$ oc adm must-gather --image=quay.io/rhdh-community/rhdh-must-gather:latest -- /usr/bin/gather --with-secrets
----

On Kubernetes (using Helm values file):
[source,yaml]
----
mustGather:
  includeSecrets: true
----
--

. Optional: Combine secrets collection with other configuration options:
+
--
On {ocp-short} (collect secrets from specific namespaces only):
[source,terminal]
----
$ oc adm must-gather --image=quay.io/rhdh-community/rhdh-must-gather:latest -- /usr/bin/gather --with-secrets --namespaces rhdh-prod
----

On Kubernetes (using Helm values file):
[source,yaml]
----
mustGather:
  namespaces: "rhdh-prod"
  includeSecrets: true
  excludeCollectors:
    - operator  # Exclude for Helm deployments
----
--

. Wait for collection to complete.
Collection with secrets takes the same amount of time as basic collection, as the tool performs redaction during collection.

. Locate the redaction report in the output directory:
+
[source,terminal]
----
$ find must-gather.local.<timestamp> -name "redaction-report.txt"  # On OpenShift
$ find must-gather-output -name "redaction-report.txt"             # On Kubernetes
----

. Review the redaction report to understand what the tool redacted:
+
[source,terminal]
----
$ cat must-gather.local.<timestamp>/redaction-report.txt  # On OpenShift
$ cat must-gather-output/redaction-report.txt             # On Kubernetes
----
+
The report lists the secret names and fields the tool redacted, along with the redaction patterns the tool applied.

. Manually review collected secrets before sharing with support:
+
[source,terminal]
----
$ find must-gather.local.<timestamp> -path "*/secrets/*" -name "*.yaml"  # On OpenShift
$ find must-gather-output -path "*/secrets/*" -name "*.yaml"             # On Kubernetes
----
+
Open the secret files and verify that:
+
. Sensitive values show redaction markers like `[REDACTED]` or `***`
. No actual passwords, tokens, or API keys appear in the files
. The tool redacts or truncates certificate data
+
If you find any residual sensitive values that automatic redaction did not catch, manually redact them before uploading.

. Optional: Search for environment-specific patterns that default redaction might not cover:
+
[source,terminal]
----
$ grep -r "mycompany.com" must-gather-local.<timestamp>/  # On OpenShift
$ grep -r "internal.network" must-gather-output/          # On Kubernetes
----
+
Search for patterns that identify your organization or internal infrastructure that attackers could use for social engineering or targeted attacks.
Internal service hostnames only visible within the cluster (for example, `backstage.rhdh-prod.svc.cluster.local`) typically do not need redaction.
Redact domain names, external hostnames, or identifiers that attackers could use to identify your organization or facilitate attacks.

. Document what you included in your support case description:
+
When attaching the `must-gather` output to your support case, note in the case description that you included secrets and that you have reviewed the redaction.
This helps support engineers understand the available data and ensures proper handling.

.Verification

. Verify that secret files are present in the output:
+
[source,terminal]
----
$ find must-gather.local.<timestamp> -path "*/secrets/*" -type f | wc -l  # On OpenShift
$ find must-gather-output -path "*/secrets/*" -type f | wc -l             # On Kubernetes
----
+
The output includes at least one secret file for each namespace where {product-very-short} stores configuration data.

. Verify the tool generated the redaction report:
+
[source,terminal]
----
$ cat must-gather.local.<timestamp>/redaction-report.txt | head -20  # On OpenShift
$ cat must-gather-output/redaction-report.txt | head -20             # On Kubernetes
----

. Spot-check a few secret files to confirm redaction:
+
[source,terminal]
----
$ cat must-gather.local.<timestamp>/namespaces/rhdh-prod/secrets/postgres-credentials.yaml
----
+
The file includes keys like `password`, `username`, `host` with values marked as `[REDACTED]` or similar.

.Troubleshooting

If the tool does not collect secrets::
Verify you have cluster administrator permissions or appropriate `RBAC` to read secrets.
The must-gather pod requires elevated privileges to access secrets across namespaces.

If the redaction report is missing::
The tool generates the report only when you use the `--with-secrets` flag.
Verify you enabled secrets collection in your `must-gather` command or Helm values.

If sensitive data appears in the output::
The automatic redaction uses pattern matching and might not catch all custom or environment-specific sensitive data.
Always manually review the output before sharing.
If you find unredacted sensitive data, redact it manually or regenerate secrets after the support case is resolved.
